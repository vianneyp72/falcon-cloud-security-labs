1. Intro & Architecture

~5 min | Beginner

The crowdstrike.falcon Ansible collection is CrowdStrike's official automation toolkit for sensor lifecycle management. Instead of manually SSHing into each host to copy and install packages, the collection talks directly to the Falcon API to download the correct sensor for each target OS, install it, configure the CID and tags, and start the service -- all in one playbook run.

┌────────────────────────────────────────────────────────────────────────┐ │ Your Workstation (Ansible Control Node) │ │ │ │ ansible-playbook deploy-falcon.yml │ │ ┌────────────────────────────────────────────────────────────────────┐ │ │ │ crowdstrike.falcon.falcon_install │ │ │ │ - Authenticates to Falcon API (OAuth2) │ │ │ │ - Downloads correct .deb for target OS │ │ │ │ - Installs the package │ │ │ ├────────────────────────────────────────────────────────────────────┤ │ │ │ crowdstrike.falcon.falcon_configure │ │ │ │ - Sets CID (auto-fetched from API) │ │ │ │ - Applies grouping tags │ │ │ │ - Starts falcon-sensor service │ │ │ └──────────┬────────────────────────────────┬──────────────────────┘ │ └─────────────┬────────────────────────────────┬─────────────────────────┘ │ SSH │ SSH ▼ ▼ ┌────────────────────────┐ ┌────────────────────────┐ │ falcon-linux-deb-12 │ │ falcon-linux-deb-13 │ │ (Debian 12 Bookworm) │ │ (Debian 13 Trixie) │ └────────────┬───────────┘ └────────────┬───────────┘ │ │ └──────────┬─────────────────────┘ │ HTTPS/443 ▼ ┌───────────────────────┐ │ CrowdStrike Cloud │ └───────────────────────┘

Key differences from bare Ansible:

  • No need to download .deb files manually -- the collection fetches them via API
  • No OS-specific branching -- the roles detect the OS automatically
  • CID is auto-fetched from the API -- you don't need to copy it from the console
  • Idempotent -- re-running won't reinstall if the sensor is already present

2. Prerequisites & Ansible Setup

~10 min | Intermediate

Step 1: Install Ansible

Ansible runs on your workstation and connects to target VMs via SSH. The crowdstrike.falcon collection also requires the FalconPy SDK on the control node to authenticate with the Falcon API.
bash
pip3 install ansible crowdstrike-falconpy
bash
ansible --version
Look for ansible [core 2.15+] -- the collection requires 2.15 or newer.

Step 2: Install the crowdstrike.falcon collection

The collection provides the falcon_install and falcon_configure roles that handle the entire sensor lifecycle. Without it, you'd write all the installation logic yourself.
bash
ansible-galaxy collection install crowdstrike.falcon
Look for: crowdstrike.falcon:<version> was installed successfully

Step 3: Create a Falcon API Client

The collection authenticates to the Falcon API to download the sensor installer and fetch your CID. You need an API client with the right scopes.
The secret is only shown once. Copy it immediately. If you lose it, you'll need to create a new client.
CLI equivalent (verify credentials via FalconPy)
bash
python3 -c "
from falconpy import OAuth2
auth = OAuth2(client_id='<YOUR_CLIENT_ID>', client_secret='<YOUR_CLIENT_SECRET>')
print(auth.token()['status_code'])  # Should print 201
"

3. Create 2 GCE VMs

~10 min | Intermediate

Step 1: Create the Debian 12 VM

These are your target hosts where the Falcon sensor will be deployed. We create them in the GCP Console first for muscle memory, then import them into Terraform later.
CLI equivalent
bash
gcloud compute instances create falcon-linux-deb-12 \
  --machine-type=e2-medium \
  --image-family=debian-12 \
  --image-project=debian-cloud \
  --tags=falcon-lab \
  --zone=us-central1-a

Step 2: Create the Debian 13 VM

CLI equivalent
bash
gcloud compute instances create falcon-linux-deb-13 \
  --machine-type=e2-medium \
  --image-family=debian-13 \
  --image-project=debian-cloud \
  --tags=falcon-lab \
  --zone=us-central1-a

Step 3: Lock down SSH to your IP only

GCP's default VPC includes a default-allow-ssh rule that allows port 22 from 0.0.0.0/0. We'll add firewall rules scoped to our falcon-lab tag that allow SSH only from your IP.
bash
curl -s ifconfig.me
CLI equivalent
bash
MY_IP=$(curl -s ifconfig.me)

gcloud compute firewall-rules create falcon-lab-allow-ssh \
  --network=default \
  --priority=1000 \
  --direction=INGRESS \
  --action=ALLOW \
  --rules=tcp:22 \
  --source-ranges="${MY_IP}/32" \
  --target-tags=falcon-lab

gcloud compute firewall-rules create falcon-lab-deny-ssh \
  --network=default \
  --priority=1001 \
  --direction=INGRESS \
  --action=DENY \
  --rules=tcp:22 \
  --source-ranges=0.0.0.0/0 \
  --target-tags=falcon-lab
GCP evaluates firewall rules by priority (lower = higher). Your IP matches the allow rule at 1000 first. All others hit the deny rule at 1001.

Step 4: Verify both are running

Step 5: Initialize SSH keys

Ansible connects via SSH. Running gcloud compute ssh once per VM pushes your SSH key to GCE metadata.
bash
gcloud compute ssh falcon-linux-deb-12 --zone=us-central1-a --command="echo connected"
gcloud compute ssh falcon-linux-deb-13 --zone=us-central1-a --command="echo connected"
Look for: connected printed for each.

4. Write the Ansible Inventory & Playbook

~15 min | Intermediate

Step 1: Create the project structure

A clean Ansible project separates inventory (which hosts), playbook (what to do), and variables (secrets). This structure scales from 2 VMs to 2,000.
bash
cd ~/projects/falcon-sensor-installs/vm-linux/falcon-ansible-gce
mkdir -p group_vars/all

Step 2: Write the inventory

The inventory tells Ansible which hosts to target and how to connect. We group by OS version so we can apply different tags per group later.
ini
[debian12]
falcon-linux-deb-12 ansible_host=<EXTERNAL_IP_1>

[debian13]
falcon-linux-deb-13 ansible_host=<EXTERNAL_IP_2>

[falcon_hosts:children]
debian12
debian13

[falcon_hosts:vars]
ansible_user=<YOUR_GCP_USERNAME>
ansible_ssh_private_key_file=~/.ssh/google_compute_engine
ansible_ssh_common_args=-o StrictHostKeyChecking=no

Step 3: Encrypt API credentials with Ansible Vault

API secrets should never be in plaintext. Ansible Vault encrypts them at rest so the file is safe to commit to version control.
bash
ansible-vault create group_vars/all/vault.yml
yaml
vault_falcon_client_id: "<YOUR_CLIENT_ID>"
vault_falcon_client_secret: "<YOUR_CLIENT_SECRET>"
Remember your vault password. You'll need it every time you run the playbook.

Step 4: Create the variable reference file

This unencrypted file maps vault variables to the names the collection expects. It separates "what variables exist" (visible) from "what their values are" (encrypted).
yaml
---
falcon_client_id: "{{ vault_falcon_client_id }}"
falcon_client_secret: "{{ vault_falcon_client_secret }}"
falcon_sensor_version_decrement: 2
We use falcon_sensor_version_decrement: 2 to install N-2 (two versions behind latest) -- CrowdStrike's recommended practice for production stability.

Step 5: Write the playbook

This is the main deployment file. It calls two roles from the collection: falcon_install (downloads + installs) and falcon_configure (sets CID, tags, starts the service).
yaml
---
- name: Deploy CrowdStrike Falcon Sensor
  hosts: falcon_hosts
  vars:
    falcon_tags: "GCP/us-central1,Lab,ansible-deployed"

  roles:
    - role: crowdstrike.falcon.falcon_install
      vars:
        falcon_sensor_version_decrement: 2

    - role: crowdstrike.falcon.falcon_configure
      vars:
        falcon_tags: "{{ falcon_tags }}"
Do NOT add become: true at the play level. The roles handle privilege escalation internally.

Step 6: Test connectivity

bash
ansible -i inventory.ini falcon_hosts -m ping
Look for: SUCCESS with "pong" for both hosts.

5. Deploy the Sensor

~10 min | Intermediate

Step 1: Run the playbook

This single command deploys the Falcon sensor to both VMs. The collection authenticates to the Falcon API, downloads the correct package for each OS version, installs it, sets the CID, applies tags, and starts the service.
bash
ansible-playbook -i inventory.ini deploy-falcon.yml --ask-vault-pass
Look for the PLAY RECAP: failed=0 across both hosts.

Step 2: Verify idempotency

Running the same playbook again should show mostly ok (green) instead of changed (yellow). This proves the collection won't reinstall a sensor that's already present.
bash
ansible-playbook -i inventory.ini deploy-falcon.yml --ask-vault-pass
Most tasks should show ok. The collection skips installation if the sensor is already at the correct version.

6. Verify in Falcon Console

~5 min | Intermediate

Step 1: Confirm hosts registered

The sensor phones home to CrowdStrike cloud after starting. Hosts should appear within 2-5 minutes.
Confirm: Status Online, Platform Linux, OS Debian 12/13, Sensor version N-2, Tags: GCP/us-central1,Lab,ansible-deployed

Step 2: Verify tags

Hosts not appearing? Check VPC firewall allows egress to *.cloudsink.net:443. Run: ansible ... -m command -a "systemctl status falcon-sensor" --become

7. Connect Back to Terraform

~15 min | Intermediate

You've built everything by hand -- now let's make it repeatable. Import your existing GCE VMs into Terraform so you can tear down and recreate with one command.

Terraform manages infrastructure (VMs, network). Ansible manages configuration (sensor deployment). Together: terraform apply creates the VMs, then ansible-playbook deploys the sensor.

Step 1: Initialize Terraform

bash
cd ~/projects/falcon-sensor-installs/vm-linux/falcon-ansible-gce
terraform init

Step 2: Fill in terraform.tfvars

hcl
project_id   = "<YOUR_GCP_PROJECT_ID>"
region       = "us-central1"
zone         = "us-central1-a"
machine_type = "e2-medium"
my_ip_cidr   = "<YOUR_PUBLIC_IP>/32"  # Run: curl -s ifconfig.me

Step 3: Import existing resources

terraform import tells Terraform "this resource in my .tf file corresponds to this real VM that already exists." After import, Terraform tracks it in state.
bash
terraform import google_compute_instance.falcon_linux_deb_12 \
  projects/<YOUR_PROJECT>/zones/us-central1-a/instances/falcon-linux-deb-12

terraform import google_compute_instance.falcon_linux_deb_13 \
  projects/<YOUR_PROJECT>/zones/us-central1-a/instances/falcon-linux-deb-13

terraform import google_compute_firewall.falcon_lab_allow_ssh \
  projects/<YOUR_PROJECT>/global/firewalls/falcon-lab-allow-ssh

terraform import google_compute_firewall.falcon_lab_deny_ssh \
  projects/<YOUR_PROJECT>/global/firewalls/falcon-lab-deny-ssh

Step 4: Validate with terraform plan

bash
terraform plan
Look for: No changes. Your infrastructure matches the configuration.

Step 5: Test the full lifecycle

bash
terraform destroy
terraform apply
ansible-playbook -i inventory.ini deploy-falcon.yml --ask-vault-pass
You now have a fully repeatable workflow: terraform apply > ansible-playbook > sensors deployed.

8. Challenges

Challenge 1: Per-Group Tags via Inventory Variables

Your team runs web servers on Debian 12 and databases on Debian 13. They want each group to automatically get different sensor grouping tags without modifying the playbook.

Hint

Create group_vars/debian12/falcon.yml and group_vars/debian13/falcon.yml with different falcon_tags values. Ansible merges group variables automatically.

Solution
bash
mkdir -p group_vars/debian12 group_vars/debian13

group_vars/debian12/falcon.yml:

yaml
falcon_tags: "GCP/us-central1,Production,WebTier"

group_vars/debian13/falcon.yml:

yaml
falcon_tags: "GCP/us-central1,Production,DatabaseTier"

Remove falcon_tags from the play-level vars in deploy-falcon.yml and re-run.

Challenge 2: Use a Sensor Update Policy Instead of Version Decrement

Your security team manages sensor versions via Falcon Sensor Update Policies. Use the policy name to determine which version to install instead of hardcoding falcon_sensor_version_decrement.

Hint

The falcon_install role accepts falcon_sensor_update_policy_name as a variable. Set it to the exact policy name from your Falcon console.

Solution
yaml
---
- name: Deploy CrowdStrike Falcon Sensor
  hosts: falcon_hosts
  roles:
    - role: crowdstrike.falcon.falcon_install
      vars:
        falcon_sensor_update_policy_name: "Production Linux - N-2"

    - role: crowdstrike.falcon.falcon_configure
      vars:
        falcon_tags: "{{ falcon_tags }}"

Challenge 3: Dynamic Inventory with falcon_hosts Plugin (Stretch)

Use the crowdstrike.falcon.falcon_hosts inventory plugin to dynamically pull hosts from the Falcon console based on tags.

Hint

Create a file ending in falcon.yml with the plugin config. Use filter with FQL to target hosts by tag.

Solution

Create inventory_falcon.yml:

yaml
---
plugin: crowdstrike.falcon.falcon_hosts
client_id: "{{ lookup('env', 'FALCON_CLIENT_ID') }}"
client_secret: "{{ lookup('env', 'FALCON_CLIENT_SECRET') }}"
filter: "tags:'GCP/us-central1'"
hostnames:
  - hostname
compose:
  ansible_host: external_ip
bash
export FALCON_CLIENT_ID="<your-id>"
export FALCON_CLIENT_SECRET="<your-secret>"
ansible-inventory -i inventory_falcon.yml --graph
